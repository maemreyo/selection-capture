# Roadmap 2026-2027 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship the roadmap on the critical path to `v1.0.0`: Windows beta, Linux alpha, async support, performance hardening, and v1 stabilization without breaking the existing sync API.

**Architecture:** Keep `capture(...)` as the stable sync core, but remove macOS-specific assumptions from the public strategy model so every platform plugs into the same engine. Add new platform implementations and advanced capabilities behind feature flags, validate them through CI and fixture-based tests first, and only then expose new APIs and docs.

**Tech Stack:** Rust 2021, Cargo, GitHub Actions, `windows` crate, Linux accessibility bindings (`atspi`/X11 helpers), optional `tokio`, existing macOS accessibility crates, `make ci`, `cargo test --all-features`

---

## Confirmed Roadmap

1. **Phase 1 is the real critical path.** `docs/technical/ROADMAP_2026_2027.md` and `docs/technical/CROSS_PLATFORM_TESTING.md` both point to Windows beta first, Linux alpha second, and CI expansion immediately after.
2. **Phase 2 is additive, not foundational.** Async/performance should land after cross-platform seams exist, not before.
3. **Phase 3 is v1-adjacent differentiation.** Real-time monitoring is valuable, but only after the core platform matrix is stable.
4. **Phase 4-5 are backlog, not release blockers.** Framework integrations, plugins, enterprise modes, and telemetry should not block `v1.0.0`.

## Confirmed Technical Issues

1. `src/types.rs` hard-codes macOS semantics into the public API (`ClipboardBorrowAppleScript`, AX-specific retry buckets), which blocks clean Windows/Linux parity.
2. `src/lib.rs` only exports `MacOSPlatform`, so the current public surface does not actually represent a cross-platform crate.
3. `src/macos.rs` maps half the strategy enum to `Unavailable`, which is acceptable for one platform but not as the long-term shared method model.
4. `.github/workflows/ci.yml` does not run Windows tests and only does a Linux `cargo check`, so roadmap claims are ahead of the verification pipeline.
5. `README.md` has API drift from the real codebase, which will erode confidence while the platform surface is changing.

## Release Ordering

1. `v0.2.0`: platform-neutral strategy model + Windows beta + Windows CI
2. `v0.3.0`: Linux alpha + Linux functional CI/dev workflow
3. `v0.4.0`: optional async API + non-blocking wrappers + perf guardrails
4. `v1.0.0`: monitoring API + docs cleanup + stabilization pass
